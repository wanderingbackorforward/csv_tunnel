# 项目总包装材料

## 一句话定位

面向盾构/TBM 设备运行数据的本地 CLI 诊断工具，将持续导出的 CSV 文件自动转化为结构化异常事件报告。

---

## 简历项目描述（约 130 字）

针对盾构设备无法直接接入实时数据流的场景，开发了一套本地 CLI 诊断工具。工具覆盖从 CSV 加载、字段映射、清洗容错、时序特征提取，到规则检测、事件分段、证据提取、模板解释、多格式导出的完整链路。引入工况状态识别层（停机/低负载/正常推进/重载推进），使异常解释从"数值超阈值"升级为"工况语义"。支持 watch 目录自动处理新文件、外部 YAML 配置调参，已在真实 TBM 数据（8.6 万行）上完成验证。全程无 LLM 依赖，规则可解释、可调整。

---

## 核心技术亮点

**① 事件驱动的异常建模**
不以单个数据点为输出单元，而是将连续命中的异常点合并为事件段（含起止时间、持续时长、峰值/均值分），并通过 `min_event_points` 参数过滤短时尖峰，降低误报率。这是和简单阈值报警的本质区别。

**② 工况状态识别层**
在特征计算之后、证据提取之前，对每行打上工况标签，再将事件窗口内的主导状态注入解释层。同一个"低效掘进"事件，发生在停机状态和重载推进状态下，解释含义完全不同。

**③ 容错优先的数据处理链路**
字段映射支持别名匹配，缺失字段自动跳过对应检测规则而不报错；IQR 尖峰清洗支持按字段豁免；重采样、填充方式、最大填充步数均可外部配置。设计目标是"给什么数据都能跑，不崩"。

**④ 检测与解释严格分离**
`detector.py` 只输出布尔标志和分数，不含任何文字；`explainer.py` 只做模板渲染，不含检测逻辑。两层之间通过 `EventEvidence` dataclass 传递，可以独立替换任意一层（比如未来把模板解释换成 LLM）。

**⑤ 完整的工程链路闭环**
从 CSV 进、到 JSON / Markdown / events.csv 出，加上 watch 目录自动处理，整个流程可以无人值守运行。不是一个分析脚本，是一个可以交付给现场工程师使用的工具。

## 和传统 BI 的本质区别

传统 BI 的核心是"展示数据"——把数值画成图表，让人去看、去判断。

这个工具的核心是"理解数据"——自动识别工况、检测异常、合并事件、提取证据、生成解释，输出的是"第 3 号事件是疑似掘进阻力异常，发生在重载推进状态下，刀盘转矩均值 3,447 kNm，建议复核地质记录"，而不是一张折线图。

| | 传统 BI | 本工具 |
|---|---|---|
| 输出单元 | 图表/数值 | 事件 + 解释 |
| 需要人做什么 | 看图判断 | 核实建议 |
| 工况感知 | 无 | 有（4 类状态） |
| 可离线运行 | 依赖服务 | 本地 CLI |
| 数据源要求 | 通常需要数据库 | 直接读 CSV |

## 为什么不是"套壳脚本"

套壳脚本的特征是：写死路径、写死阈值、跑一次出一个结果、换个数据就崩。

这个工具做了以下工程决策，使它不是：

- **模块边界清晰**：9 个模块各司其职，ingestion 不做清洗，detector 不做解释，任何一层可以单独替换或测试。
- **配置外置**：所有阈值在 YAML 里，不改代码就能调参，现场工程师可以自己调。
- **容错设计**：缺字段不崩、时间戳异常不崩、编码自动识别、分隔符自动识别。这些都是真实数据里会遇到的问题，不是假设。
- **真实数据验证**：在 8.6 万行真实 TBM 数据上跑通，发现并处理了重复时间戳、IQR 误杀高负载转矩、孤立尖峰误报等真实问题，并针对性地做了设计调整（`iqr_exempt_fields`、`min_event_points=5`）。
- **可交付**：有 watch 模式、有多格式导出、有 README，不是只有作者自己能跑的脚本。

工程价值不在于代码量，在于这些决策背后的判断——知道什么该做、什么不该做、为什么这样设计。
