# 系统能回答什么问题

## 背景约束

这套系统是在"对方设备管理系统只能导出 CSV 文件"的现实约束下构建的。没有实时数据接口，没有数据库，不允许改动原有系统。所有分析都基于离线 CSV / XLS 文件，作为外挂式分析层运行。

---

## 系统能回答的问题

### 单文件层面

**1. 这个文件里有没有异常？**

`detect` 子命令对单个 CSV / XLS 文件跑完整检测链路，输出：
- 是否存在异常事件，共几个
- 每个事件的类型、时间范围、持续时长、严重度
- 每个事件的关键证据（均值、峰值、方向）
- 工程师可读的解释文本（总结 + 可能原因 + 建议关注项）

**2. 这个异常发生在什么工况下？**

工况状态识别层（`state_engine.py`）对每行数据分类为停机/低负载/正常推进/重载推进，并把事件窗口内的主导状态注入解释输出。工程师拿到的不是"推进速度偏低"，而是"该事件主要发生在重载推进状态下，推进速度偏低，疑似重载工况下掘进阻力异常"。

**3. 这个"低效"是停机导致的，还是推进中真的低效？**

这是 semantic layer 解决的核心问题。系统基于 `(event_type, dominant_state)` 的组合规则，把事件重分类为：
- **停机片段**：低效掘进事件 + 停机工况 → 属于停机/静止，不是推进参数问题
- **推进中低效掘进**：低效掘进事件 + 正常/重载推进工况 → 真实的推进效率不足
- **重载推进下的掘进阻力异常**：掘进阻力事件 + 重载推进工况 → 重载工况下的阻力异常

这个区分在真实数据中非常重要：某个高风险文件的 217 个事件中，209 个是停机片段（累计 7.2h），只有 8 个是推进中低效掘进。如果不做这个区分，AI 总结会输出"需要调整推进参数"——这对停机片段完全无意义。

**4. 这些事件合在一起说明了什么？**

LLM 总结层（`summarizer.py`）把所有事件的结构化摘要传给 LLM，生成跨事件归纳：整体评估、主要风险（3~5 条）、建议关注项（3~5 条）。LLM 的输入包含 semantic 分类统计，明确区分停机问题和推进效率问题。

---

### 批量文件层面

**5. 这批历史文件里，哪些最值得关注？**

`scan` 子命令对全量文件跑规则诊断，生成 `scan_index.csv`，包含每个文件的 `risk_rank_score`（综合事件数量、严重度、峰值分数）、`event_count`、`max_severity_label`。工程师可以直接按 `risk_rank_score` 排序，找出最值得关注的文件。

**6. Top N 高风险文件的核心问题是什么？**

`review` 子命令对 Top N 文件调 LLM 深度复核，生成 `review_summary.md`，包含：
- 每个文件的语义事件分布（停机片段 X 个/Xh，推进中低效掘进 Y 个/Yh，重载阻力异常 Z 个/Zh）
- 每个文件的 AI 总结（明确区分停机问题和推进效率问题）
- 跨文件语义分析表（各业务语义类型的汇总统计）
- 核心问题判断（如"停机时间过长是核心问题，而非推进参数偏差"）

**7. 这批文件的共性问题是什么？**

`review_summary` 的跨文件语义分析会汇总：
- 停机片段总数和总时长（跨所有 review 文件）
- 推进中低效掘进总数和总时长
- 重载推进下掘进阻力异常总数
- 主导问题判断（停机管理 / 推进效率 / 重载阻力）

## 系统不能回答的问题

诚实地说，这套系统有明确的边界：

- **不能做实时监控**：基于离线 CSV，没有实时数据流接入。
- **不能做预测**：没有历史基线对比，不能预测"下一个小时会不会出问题"。
- **不能替代现场判断**：系统输出的是"需结合施工日志确认停机原因"，不是"停机原因是换刀"——后者需要人工核实。
- **检测规则基于经验阈值**：尚未经过大量真实数据的系统性校准，可能存在误报或漏报。
- **字段映射依赖固定列名**：CSV 列名变更需要更新 `schema.py`。

---

## 为什么这套架构适合处理大量历史文件

**两阶段设计的核心价值**：scan 是纯规则，速度快，可以并行处理几十个文件，产出可信的风险排序；review 是 AI 深度分析，成本高，只用在真正值得看的文件上。这个设计让"全量覆盖 + 重点深入"成为可能，而不是在"全量 AI（成本不可控）"和"全量规则（缺少语义归纳）"之间二选一。

**semantic layer 的核心价值**：在不改动检测器的前提下，修正了"停机片段被误报为低效掘进"的语义问题。这个修正发生在数据层，而不是 prompt 层——不是靠 prompt 里写"请注意停机和推进的区别"，而是在输入给 LLM 之前就把标签改对了。这让 AI 总结的结论质量有了结构性提升，而不是依赖 LLM 自己去推断。
