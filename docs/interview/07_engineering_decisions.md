# 关键工程决策总结

## 1. 为什么先做 CLI，不先做前端

**为什么这么做：** CLI 的开发和调试成本远低于前端，而且在链路还没稳定的阶段，前端只会增加噪音。CLI 可以直接用命令行验证每一层的输出，出了问题立刻定位。

**替代方案：** 先做 Web 前端或 Streamlit dashboard，让输出更直观。

**为什么没选：** 前端需要先确定数据结构和交互逻辑，但在项目早期这些都还在变。如果先做前端，每次调整检测逻辑都要同步改前端，维护成本很高。而且现场工程师的第一需求是"能跑、能出报告"，不是"界面好看"。CLI + 导出 Markdown/CSV 已经满足这个需求。

---

## 2. 为什么先做规则检测，不先做 LLM Agent

**为什么这么做：** 工业时序数据是高频数值序列，规则检测对这类数据天然适合：确定性强、可回溯、可解释、不依赖外部服务。

**替代方案：** 直接用 LLM 分析 CSV，或者搭 ReAct Agent 做多步推理。

**为什么没选：** 原始时序数据直接喂 LLM 有两个问题：一是上下文长度限制，几万行数据根本放不进去；二是 LLM 对数值序列的判断不稳定，同样的数据跑两次可能结论不同，工业场景不能接受这个。规则检测先把"哪里有问题"确定下来，后续如果要加 LLM，可以只把事件摘要喂给它，而不是原始数据。

---

## 3. 为什么以"事件"为核心，而不是单个数据点

**为什么这么做：** 单个数据点的异常可能是噪声，也可能是传感器抖动。工程师真正关心的是"这台机器在某段时间内持续出现了什么问题"，而不是"第 1,234 行的转矩超了阈值"。

**替代方案：** 直接输出每个异常点的告警列表。

**为什么没选：** 告警列表在高频数据下会产生大量重复告警，工程师根本看不过来。把连续命中的点合并成事件，加上起止时间、持续时长、峰值分，才是可操作的信息单元。这也是后续证据提取和解释生成的基础——证据是针对事件窗口统计的，不是针对单点的。

---

## 4. 为什么要加 config，而不是把阈值写死

**为什么这么做：** 不同地质条件、不同设备型号、不同施工阶段，合理的阈值范围差异很大。写死阈值意味着换一个项目就要改代码。

**替代方案：** 把阈值写在代码里，用常量或配置字典。

**为什么没选：** 写在代码里意味着调参需要改代码、重新部署，现场工程师没办法自己调。外部 YAML 配置让调参变成一个独立操作，不需要懂代码，也不会引入代码风险。同时 config 也是文档——看 YAML 就能知道当前用的是什么阈值，不需要去翻代码。

---

## 5. 为什么用 min_event_points 解决短时尖峰误报，而不是调大 IQR

**为什么这么做：** 短时尖峰误报的根本原因是：3 个点的孤立命中也会被分段成一个事件。提高 `min_event_points` 从分段层过滤，不影响检测层的灵敏度，也不影响长时真实异常的检出。

**替代方案：** 全局调大 IQR 倍数，让清洗层更激进地去除尖峰；或者调高检测阈值。

**为什么没选：** 调大 IQR 会把真实的高负载数据也清洗掉——实验验证过，cutter_torque_kNm 在高负载工况下的高值是真实数据，不是噪声，不应该被清洗。调高检测阈值会降低对真实异常的灵敏度。`min_event_points` 是在分段层做过滤，只影响"持续时间太短的事件"，不影响检测灵敏度，是最小侵入性的解法。

---

## 6. 为什么状态层是最值得做的下一步升级

**为什么这么做：** 没有状态层时，系统对所有工况用同一套阈值，停机状态下的"低速"和正常推进中的"低速"会被同等对待，误报率高。状态层让系统先理解上下文，再解释异常，是从"数值判断"到"工况语义"的关键一步。

**替代方案：** 继续调整阈值，或者为不同时段手动标注工况。

**为什么没选：** 手动标注不可持续，调阈值治标不治本。状态层可以自动运行，复用 feature_engine 已有的 rolling mean 列，不引入新依赖，实现成本低，但对解释质量的提升是结构性的。它也为后续接入 LLM 做了铺垫——有了工况标签，可以只把"重载推进状态下的异常事件"喂给 LLM 做深度分析，而不是把所有数据都喂进去。
