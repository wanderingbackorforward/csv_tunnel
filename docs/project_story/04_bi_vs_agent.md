# 为什么这不是普通 BI，为什么这是真 Agent

---

## 问题一：为什么这不是普通 BI？

### 短版

普通 BI 是"你告诉它看什么，它给你画图"。这个系统是"你给它原始数据，它自己决定看什么、发现什么、解释什么"。

### 展开版

普通 BI 工具（Grafana、Power BI、Tableau）的工作模式是：你预先定义指标、阈值、图表，工具负责渲染。它不会主动发现问题，不会合并相邻异常点为事件，不会判断"这个低速是停机导致的还是推进效率不足"，更不会生成"建议结合施工日志确认停机原因"这样的工程师语言。

这个系统做的事情是：

1. **自动发现**：不需要人工指定"看哪个时间段"，系统自己扫描全量数据，找出所有命中规则的时间窗口。
2. **事件合并**：把逐点命中合并为有起止时间的事件段，输出单位是"事件"而不是"数据点"。BI 工具不做这件事。
3. **工况语义**：判断每个事件发生在什么工况下（停机/推进/重载），这需要跨列的组合逻辑，不是单指标阈值。
4. **语义重分类**：基于 `(event_type, dominant_state)` 的组合规则，把"低效掘进 + 停机"重分类为"停机片段"，这是业务语义层面的判断，BI 工具没有这个概念。
5. **自然语言解释**：输出"该事件主要发生在停机/静止工况下，推进速度接近零，属于停机片段而非推进中的低效掘进，需结合施工日志确认停机原因"——这不是图表，是工程师可以直接使用的结论。

BI 是可视化工具，这是诊断系统。区别在于：BI 帮你看数据，这个系统帮你理解数据。

## 问题二：为什么这是真 Agent？

### 短版

Agent 不是"调一次 LLM 生成文本"，而是"LLM 通过工具调用主动探索数据，多轮推理后得出结论"。这个系统的 agent 模式满足这个定义。

### 展开版

很多所谓"AI 分析"的实现是：把数据拼成一段文字，发给 LLM，让它输出结论。这不是 agent，这是带数据的 prompt。

这个系统的 agent 层（`agent.py`）的工作方式是：

1. LLM 收到任务描述（"分析这个 TBM 数据文件"）和可用工具列表。
2. LLM 自主决定调用哪个工具、用什么参数：先调 `run_detect` 跑完整检测链路，再调 `get_event_details` 查看具体事件，再调 `get_state_distribution` 了解工况分布。
3. 每次工具调用返回结构化结果，LLM 基于结果决定下一步。
4. 多轮之后，LLM 生成最终诊断报告。

关键点是：**LLM 在主动探索，而不是被动接收**。它可以决定"这个事件值得深入看"，然后调工具去看；也可以决定"工况分布已经足够，不需要再查了"。这是 tool-using agent 的核心特征。

当然，这个 agent 的工具集是固定的（不是通用 agent），推理深度也有限。但它满足 agent 的基本定义：感知（工具返回）→ 决策（选择下一步工具）→ 行动（调用工具）的循环。

## 问题三：为什么没有一开始就让大模型直接看原始 CSV？

### 短版

上下文放不下，数值判断不稳定，检测结果需要确定性。这三个问题加在一起，让"LLM 直接看原始 CSV"在工程上不可行。

### 展开版

原始 TBM 数据文件是高频时序数据，单文件 8.6 万行，每行十几个数值列。直接喂给 LLM 有三个根本性问题：

**1. 上下文长度**

8.6 万行 × 15 列 = 约 130 万个数值。即使是 128K 上下文的模型，也放不下一个完整文件。截断意味着丢失信息，而异常往往在时序上有跨段关联。

**2. 数值判断不稳定**

LLM 对数值序列的判断是概率性的，不是确定性的。同样的数据，不同温度、不同 prompt 措辞，可能得出不同结论。工业诊断需要可回溯、可复现的结果——"上次说有问题，这次说没问题"在工程上不可接受。

**3. 没有领域规则的 LLM 容易产生幻觉**

LLM 不知道"刀盘转矩 3000 kNm 在重载推进工况下是正常的，但在停机状态下是异常的"。这类领域知识需要显式编码为规则，不能靠 LLM 自己推断。

**正确的分工是**：规则检测负责确定性判断（哪些时间窗口命中了哪些规则），LLM 负责语义归纳（这些事件合在一起说明了什么）。LLM 的输入是精简的结构化摘要（每个事件约 5 行文字），不是原始数值序列。这样既利用了 LLM 的语言能力，又保证了检测结果的确定性。
