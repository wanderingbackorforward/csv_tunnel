# 面试口述版项目介绍

## 1 分钟版

这个项目的背景是：我们合作的盾构施工现场，设备管理系统只能定期导出 CSV 文件，没有实时数据接口，也不允许改动原有系统。工程师每天要手动翻几十个 CSV 文件找异常，效率很低，而且容易漏掉跨时段的规律性问题。

我做的事情是：在不改动原有系统的前提下，写了一个外挂式 CLI 工具，读取这些 CSV，自动完成字段识别、数据清洗、异常检测、事件分段，最后输出工程师可以直接看的报告。

技术上有几个我觉得比较有意思的决策：

第一，检测和解释严格分离。`detector.py` 只输出布尔标记和分数，`explainer.py` 负责文本，互不耦合。这样阈值调整不会影响解释逻辑，解释模板改了也不影响检测结果。

第二，加了工况状态识别层。真实数据里停机占了 70%，停机时的"低速"和推进中的"低速"含义完全不同。我在特征层之后加了一个状态分类，把每行打上停机/正常推进/重载推进的标签，再把事件窗口内的主导状态注入解释输出。

第三，后来发现状态层还不够——大量"低效掘进"事件的主导状态是停机，这在语义上是错的。所以我加了一个 semantic layer，基于 `(event_type, dominant_state)` 的组合规则，把停机状态下的低效掘进重分类为"停机片段"，让 AI 总结不再把停机问题误写成推进参数问题。

第四，批量处理用了两阶段设计：`scan` 先对全量文件跑规则诊断，`review` 再对 Top N 高风险文件调 LLM 深度复核。这样 AI 调用成本集中在真正值得看的文件上。

最终在真实的几十个 XLS 文件上跑通了，review_summary 能正确区分"停机时间过长"和"推进效率不足"这两类完全不同的问题。

## 30 秒压缩版

这是一个针对盾构施工数据的外挂式诊断工具。背景是对方系统只能导出 CSV，不能改。我做了一条完整的时序处理链路：字段识别、清洗、规则检测、事件分段、工况状态识别、语义重分类，最后接 LLM 做跨事件归纳。

核心设计是两点：一是检测和解释分离，规则保证确定性，LLM 只做语义归纳；二是 semantic layer，把停机状态下的"低效掘进"重分类为"停机片段"，避免 AI 把停机问题误报成推进参数问题。批量处理用 scan + review 两阶段，AI 成本只花在高风险文件上。
