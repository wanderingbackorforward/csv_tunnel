# 关键工程决策

---

## 决策 1：为什么先做 CLI，不先做前端？

**问题**：有前端会更直观，为什么不先做？

**选择**：先做 CLI，前端留到后续。

**理由**：
- 核心价值在于检测逻辑和解释质量，不在于界面。CLI 能更快验证核心链路是否正确。
- 对方系统只能导出 CSV，没有实时数据流，前端的交互价值有限——工程师更需要的是"跑一个文件，得到一份报告"，而不是"实时看仪表盘"。
- CLI 的输出（JSON / Markdown / CSV）可以直接被任何前端消费，接口稳定后再做前端不会返工。

**权衡**：牺牲了演示效果，换来了更快的核心链路验证和更低的技术债。

---

## 决策 2：为什么先做规则检测，不先让模型直接分析原始时序？

**问题**：现在 LLM 很强，为什么不直接让它看数据？

**选择**：先做规则检测，LLM 只做下游归纳。

**理由**：
- 单文件 8.6 万行数值序列放不进 LLM 上下文。
- 工业诊断需要确定性和可回溯性：同样的数据必须得出同样的结论，LLM 的概率性输出不满足这个要求。
- LLM 不知道领域规则（如"重载推进下转矩 3000 kNm 是正常的"），没有显式规则的情况下容易产生幻觉。

**权衡**：规则需要人工编写和校准，有维护成本。但这个成本是可控的，而 LLM 直接判断数值的不确定性是不可控的。

## 决策 3：为什么以事件为中心，不以单点为中心？

**问题**：检测到异常点就报告，不是更简单吗？

**选择**：把连续命中点合并为事件段，输出单位是事件。

**理由**：
- 单点异常大量存在噪声（传感器抖动、采样误差），工程师无法从几千个孤立异常点中找到规律。
- 工程师关心的是"这台机器在哪个时间段出了什么问题"，而不是"第 12345 行数据超阈值"。
- 事件段有起止时间、持续时长、峰值分数，可以直接用于施工日志对照和优先级排序。

**权衡**：分段逻辑引入了 `gap_tolerance_points` 和 `min_event_points` 两个参数，需要调整。但这个复杂度是值得的——它把"数据点"变成了"工程师语言"。

---

## 决策 4：为什么要加状态层？

**问题**：调整阈值不能解决停机/推进混淆的问题吗？

**选择**：加工况状态识别层，而不是调阈值。

**理由**：
- 停机状态下的"低速"和推进中的"低速"是两种完全不同的物理含义，用同一个阈值对待它们，无论阈值怎么调都会有误报或漏报。
- 状态层是零侵入插入：不改 `detector.py` 的任何逻辑，只在 evidence 层增加一个字段。主链路不受影响。
- 状态信息注入解释输出后，工程师拿到的报告从"推进速度偏低"变成了"该事件主要发生在停机/静止状态下"，信息量质的提升。

**权衡**：状态分类规则基于经验阈值，可能在边界工况下分类不准。但四分类（停机/低负载/正常推进/重载推进）已经覆盖了主要场景，精度足够用。

---

## 决策 5：为什么要做 semantic layer？

**问题**：状态层已经有 dominant_state 了，为什么还要 semantic layer？

**选择**：在状态层之上加一个事件语义重分类层。

**理由**：
- 状态层解决的是"每行数据是什么工况"，semantic layer 解决的是"这个事件在业务上是什么问题"。
- 真实数据中，大量"低效掘进"事件的 dominant_state 是停机——这在语义上是错的，但 `detector.py` 不知道，因为它只看数值规则。
- 如果不做 semantic layer，LLM 收到的输入是"217 个低效掘进事件"，它会输出"需要调整推进参数"——这对停机片段完全无意义。
- semantic layer 的规则极简：`(event_type, dominant_state)` 的组合映射，不改 detector，保留原始 `event_type` 供回溯。

**权衡**：增加了一个新概念（semantic_event_type），需要在所有输出层（evidence / explainer / exporter / cli / reviewer / summarizer）都接入。但这个改动是增量的，不破坏现有接口。

## 决策 6：为什么 scan 先于 review？

**问题**：为什么不直接对所有文件跑 review？

**选择**：scan 先做全量规则诊断，review 再对 Top N 做 AI 复核。

**理由**：
- 全量文件（几十个 XLS）跑 LLM 成本高、速度慢，而且大部分文件可能根本没有值得深入分析的异常。
- scan 是纯规则，速度快，可以并行，产出 `scan_index.csv` 作为风险排序依据。
- review 基于 `risk_rank_score` 筛出真正高风险的文件，把 AI 调用成本集中在值得看的地方。

**权衡**：两阶段增加了流程复杂度。但这个复杂度换来了成本可控和结果可解释——scan 的结果是确定性的，review 的 AI 总结建立在可信的规则输出之上。

---

## 决策 7：为什么 review 只看 Top N，而不是全量跑 AI？

**问题**：全量跑 AI 不是更全面吗？

**选择**：默认只 review Top N（可配置，默认 5）。

**理由**：
- LLM API 调用有成本和延迟。几十个文件全量跑，每个文件几十秒，总时间不可接受。
- 大部分文件的异常事件数量少、风险低，LLM 总结的增量价值有限——规则输出已经足够。
- `risk_rank_score` 是基于事件数量、严重度、峰值分数的综合排序，Top N 已经覆盖了最值得关注的文件。
- `--top-n` 参数可以调整，用户可以根据实际需求决定覆盖范围。

**权衡**：可能漏掉排名靠后但有特殊问题的文件。但这是有意识的取舍：把有限的 AI 资源用在最高价值的地方，而不是平均分配。
