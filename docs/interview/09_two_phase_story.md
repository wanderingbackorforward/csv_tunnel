# 两阶段迭代故事

**面试问题：你是怎么规划这个项目的迭代路径的？**

---

## 第一阶段：先把链路做稳（v0.1 → v0.2）

第一阶段的目标只有一个：**在真实数据上跑通一条完整链路**。

我没有一开始就想着加 LLM 或者做 Web 界面，而是先把最核心的问题解决掉：给一个 CSV，能不能自动识别字段、清洗数据、检测异常、输出可读的事件报告。

这个阶段遇到的问题都是真实数据带来的：

- 重复时间戳、混合时间格式——清洗层要能处理，不能崩
- IQR 清洗把高负载工况下的真实高转矩值误判为尖峰——加了 `iqr_exempt_fields` 豁免机制
- 3 个点的孤立命中也会生成事件，大量噪声——加了 `min_event_points=5` 在分段层过滤

这些问题不是设计出来的，是跑真实数据（8.6 万行）时发现的，然后针对性地做了设计调整。

第一阶段结束时，系统已经能稳定运行，有 watch 模式、YAML 配置、多格式导出。这是一个可以交付的基线，不是 demo。

---

## 第一阶段末：加工况状态层（v0.2 → v0.2.x）

跑真实数据时发现一个结构性问题：机器大部分时间是停机的（真实数据里停机占 70%），停机状态下的"低速"和正常推进中的"低速"含义完全不同，但系统用同一套阈值对待它们。

这不是调阈值能解决的问题，需要在解释层之前引入工况上下文。

所以我加了状态识别层：在 feature_engine 之后、evidence 之前，对每行打上工况标签（停机/低负载/正常推进/重载推进），再把事件窗口内的主导状态注入解释输出。

这一步让输出从"液压压力波动超阈值"变成了"该事件主要发生在正常推进状态下，液压压力出现明显波动，建议检查主推进泵"。工程师拿到报告，第一眼就能判断这个事件值不值得关注。

---

## 第二阶段：在稳定基线上叠加 LLM（v0.3）

第一阶段结束后，我评估了三个升级方向：

1. **状态层 v2**（滑动窗口投票、状态机）
2. **历史基线对比**（相对异常检测）
3. **LLM 总结层**（跨事件归纳）

我选了 LLM 总结层，原因是：

- **收益最确定**：事件结构已经完整，LLM 的输入已经准备好了，不需要改任何检测逻辑
- **风险最低**：对主链路零侵入，失败时静默降级
- **切入点最合适**：LLM 擅长语义归纳，不擅长数值序列判断。"今天 3 个事件集中在上午 10 点"这种跨事件结论，规则做不了，LLM 很适合

**关键设计决策：**

LLM 只接收精简的结构化摘要（每个事件约 5 行文字），不接触原始 DataFrame。这解决了两个问题：一是上下文长度（几万行数据放不进去），二是确定性（检测结果由规则保证，不受 LLM 随机性影响）。

`--llm-summary` 是可选 flag，不加时完全不调用 LLM。失败降级分五层：无 SDK、无 API key、无事件、超时/API 报错、JSON 解析失败，全部返回 None，不向上抛异常。

---

## 两阶段的核心判断

**第一阶段的判断**：先做规则，不先做 LLM。工业时序数据需要确定性和可回溯性，规则检测天然满足，LLM 不行。

**第二阶段的判断**：LLM 的切入点是"跨事件归纳"，不是"检测"。把 LLM 放在规则检测的下游，让它做它擅长的事，而不是用它替代规则。

**贯穿两阶段的判断**：每次升级都在稳定基线上叠加，不重构主链路。状态层是零侵入插入，LLM 层是可选 flag。这让每个阶段的风险都可控，也让每个阶段的成果都可以独立交付。
