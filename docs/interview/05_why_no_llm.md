# 面试回答：为什么没先上 LLM / Agent

**问题：你为什么没有一开始就上大模型、ReAct 或多 Agent，而是先做规则和模板解释？**

---

这个决策其实挺自然的，不是刻意回避 LLM，而是当时想清楚了几件事。

首先是数据特点的问题。TBM 的运行数据是高频时序数据，一天可以有几万到十几万行，每行几十个字段。如果直接把原始数据喂给大模型，上下文会爆，而且大模型对数值序列的理解并不擅长——它更擅长处理语义，不擅长判断"这段时间转矩的滚动均值是否持续超过阈值"。

其次是稳定性和可回溯的问题。工业场景里，检测结果需要是确定性的——同样的数据，跑两次结果要一样，而且要能解释"为什么这个点被标记了"。规则检测天然满足这个要求，LLM 不行。

第三是我当时的优先级判断：先把可用链路做稳，再考虑更强的能力。如果一开始就搞 Agent，链路还没稳，调试成本会很高，而且出了问题很难定位是哪一层的问题。

所以我选择了规则检测 + 事件分段 + 模板解释这条路，先把"能用、稳定、可解释"做到，再考虑升级。

当然我不是觉得 LLM 没价值。我觉得它在这个项目里有一个很合适的切入点：**日报总结和跨事件归纳**。比如"今天发生了 5 个事件，其中 3 个集中在上午 10 点到 11 点，可能和换刀操作有关"——这种跨事件的语义归纳，规则做不了，LLM 很适合。但那是下一步的事，不是第一步。
