# 项目亮点 / 难点 / 反思

## 项目亮点

- **完整链路闭环**：从 CSV 进到报告出，覆盖字段映射、清洗、特征、检测、分段、证据、解释、导出全流程，不是只做了其中某一段。
- **事件驱动输出**：以事件段为核心输出单元，而不是告警点列表，输出结果对工程师直接可用。
- **工况状态层**：在解释层之前引入工况识别，让异常解释带上工况上下文，从"数值超阈值"升级为"工况语义"。
- **容错设计经过真实验证**：缺列、重复时间戳、混合时间格式、损坏文件均不崩溃，在 8.6 万行真实数据上跑通。
- **可交付性**：支持 watch 目录自动处理、YAML 外部配置、多格式导出，不是只有作者自己能跑的脚本。

---

## 项目难点

- **真实数据的脏数据处理**：真实 TBM 数据存在重复时间戳、混合时间格式、部分字段全为 NaN、编码不一致等问题，需要在清洗层逐一处理，且不能因为某个字段有问题就让整个流程崩溃。
- **短时尖峰误报的定位和解决**：最初 `min_event_points=3` 时，3 个点的孤立命中也会生成事件，导致大量噪声事件。需要区分"这是真实异常还是传感器抖动"，最终通过分段层过滤而不是调整检测阈值来解决，避免影响真实异常的检出率。
- **IQR 清洗与高负载数据的冲突**：全局 IQR 清洗会把高负载工况下的真实高转矩值误判为尖峰去除。需要设计 `iqr_exempt_fields` 机制，允许按字段豁免，同时通过实验验证豁免与否对最终检测结果的影响。
- **工况状态层的插入位置**：状态层需要在 feature_engine 之后（复用 rolling mean 列）、evidence 之前（让状态信息注入证据和解释），插入位置的选择影响整个数据流的设计。

---

## 项目反思与后续升级方向

- **当前阈值仍以经验值为主**：检测阈值是基于对 TBM 工程的一般认知设定的，没有基于历史数据做统计标定。如果有足够的历史正常段数据，可以用分位数或均值±N倍标准差来自动确定阈值，而不是手动设定。
- **状态层 v1 比较粗**：当前 4 类状态的判定规则是基于单一阈值的，没有考虑状态转换的连续性（比如从停机到推进的过渡段）。后续可以引入滑动窗口投票或简单的状态机来提高状态识别的稳定性。
- **没有历史基线对比**：当前每次分析是独立的，没有和历史正常段做对比。如果能建立"正常推进段的基线分布"，检测就可以从"超过固定阈值"升级为"偏离历史基线"，对设备退化等渐变异常更敏感。
- **LLM 总结层已落地，但 prompt 还有优化空间**：第二阶段已实现 `--llm-summary`，LLM 做跨事件归纳。当前 prompt 是单轮的，后续可以根据实际输出质量迭代 prompt，或者在有足够事件时做分批归纳再合并。
- **暂无 Web 前端**：当前输出是 CLI + 文件，对非技术用户不够友好。后续可以考虑轻量级 Web 界面，但优先级低于把检测和解释质量做好。
